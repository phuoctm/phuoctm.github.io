Orientation

What is Claude Code?

You've used Claude in a browser.
Claude Code is Claude in your computer.

Think pair programmer,
not search box.
  • 📂It can read & write files and run commands on your machine.
  • 💬You give it goals in plain English; it builds, runs, and explains.
  • 🎛️You stay in control: it asks before doing things, and you review results.
  • ⚙️Perfect for automation, scripts, and connecting to APIs like Qlik's.
05 / 28

The rhythm

How a session flows

  1. You type a prompt — a goal, in plain language.
  2. Claude proposes & acts — writes files, runs commands.
  3. You approve what it wants to do.
  4. It verifies the result and tells you what happened.
  5. Repeat — each prompt adds the next capability.
⚠️ Your screen won't match these slides exactly. Claude adapts to your OS, your tenant, and what it finds. If a step differs, errors, or asks a question — just answer it in plain English.

Driving Claude Code

Modes, model & effort

Mode
Ask permissions1
Accept edits2
Plan mode3
Auto mode✓4
Bypass permissions5
Auto + 🎤 Opus 4.8 High
  • Ask permissionsapproves every action. Safest — best when you want to learn all the technical details.
  • Accept editsauto-applies file edits; still asks before running commands.
  • Plan modeexplores and proposes a plan; changes nothing until you approve.
  • Auto moderuns on its own, pausing only for risky or irreversible actions. What we use today.
  • Bypass permissionsnever asks. Only when you fully trust the task.
🚀 Use the strongest model for coding — Opus 4.8 today (bottom-right). For really hard problems, also raise effort above High.

Before we start

  • 💻A laptop (Windows or Mac)
  • 🔑Local admin rights on that laptop — to install software and run npm
  • 🤖Claude Code installed, signed in with your Claude account
  • 🟢A Qlik Cloud account that can create an API key
  • 🌐Internet access

The only things you set up by hand are Claude Code and your Qlik API key. Claude handles the rest.

10 / 28

Prereq C

Get your Qlik API key

  1. Click your avatar (top-right) → Profile settings.
  2. API keys → Generate new key → copy it now (shown only once).
  3. Note your tenant hostname from the URL, e.g. your-tenant.eu.qlikcloud.com.
💡 Don't see "API keys"? A Qlik admin needs to enable them for your tenant.

Part 2

The morning reload watchdog

16 / 28

Step 3

Build the watchdog

▸ Prompt
Create a skill I can run every morning that checks my scheduled reloads
since it last ran and flags anything off — failed reloads, tasks Qlik
auto-disabled after repeated failures, big jumps or drops in app size,
and reloads that didn't run when they should. Keep the report to current
issues only, and draft me an email summary.

What Claude doesBuilds the skill, keeps a small local history so it can compare day-to-day, and runs once to set a baseline.

17 / 28

Step 4

Run the watchdog

▸ Prompt
Run the reload watchdog.

…plus an email draft in your inbox to review and send. It shows current issues only — no pile-up of old noise.

Expecta short, prioritized report —

  • 🔴High — e.g. a task auto-disabled after failures
  • 🟠Medium — e.g. a reload that didn't run
  • ✅Everything else healthy

Part 3

The load-script debugger

20 / 28

Step 5

Build the debugger

▸ Prompt
Create a skill to debug a Qlik load script. I give it an app ID; pull
that app's reload log. If the last reload failed, tell me when it ran,
how long it took, the likely cause, and the exact failing command. If it
succeeded, compare it with the previous successful run — rows and time
per table. Add an option to only analyze the latest run for performance.

ResultOne skill with three modes — diagnose a failure, compare runs, or performance review.

Step 6

Use it

Find an app ID (copy it from the Qlik app's URL, or ask Claude to list your apps).

To debug a reload

▸ Prompt
Debug the reload for app <paste-app-id>.

→ when it ran, how long, likely cause, the exact failing command, log link.

For performance ideas only

▸ Prompt
Only analyze the latest reload of app
<paste-app-id> for performance.

→ slowest steps + tips (e.g. push joins/filters to the source database).
